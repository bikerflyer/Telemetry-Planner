# ROV Telemetry Planner Features

This file documents the route-planning features in the planner, with emphasis on the routing screen, route waypoint options, route leg options, and movement modes.

## Routing Screen Overview

The routing screen is used to build and edit the tender or vehicle route that can be exported from the planner. A route is made up of:

- `Route Waypoints`
  - Individual points the route will visit in order.
- `Route Legs`
  - The line segment between one route waypoint and the next.
- `Route Properties`
  - Navigation settings applied either to a waypoint or to a leg.
- `Pattern Routing`
  - Spiral or lawnmower route patterns inserted into the route.
- `Boundary Avoidance`
  - Boundary legs or circles that the route tries to avoid or work around.

## Route Waypoint Options

Open `Route Waypoint Properties` by right-clicking a route waypoint.

These settings apply to a waypoint itself:

- `Name`
  - Optional label for the waypoint.
- `Target Speed (m/s)`
  - Desired speed approaching or passing that waypoint.
- `Turn Radius (m)`
  - Requested turning radius around that waypoint.
- `Depth (m)`
  - Absolute target depth at the waypoint.
- `Depth From Bottom (m)`
  - Alternative vertical control if operating relative to bottom rather than absolute depth.
- `On Arrival Action`
  - What should happen when the waypoint is reached.
- `Hold Time (sec)`
  - Time to wait or dwell if the arrival behavior requires a stop or hold.
- `Cross Track Tol (m)`
  - Allowed route tracking error before correction is needed.
- `Heading Mode`
  - How vehicle heading should be managed near that waypoint.
- `Notes`
  - Free-text operator notes.

Waypoint properties are exported inside the waypoint `nav` object in the route payload.

## Route Leg Options

Open `Route Leg Properties` by right-clicking a route leg.

These settings apply to the line between two route waypoints:

- `Name`
  - Optional label for the leg.
- `Target Speed (m/s)`
  - Desired travel speed on that leg.
- `Turn Radius (m)`
  - Useful where the vehicle needs a softer turn into or out of a leg.
- `Hold Time (sec)`
  - Optional timing behavior associated with the leg.
- `Cross Track Tol (m)`
  - Lateral tracking tolerance while following that leg.
- `Heading Mode`
  - Heading behavior while travelling the leg.
- `Movement Mode`
  - The leg motion profile.
- `Notes`
  - Free-text notes for that leg.

Leg properties are exported separately in `route_leg_properties`.

## Waypoint Roles

Each route waypoint carries a `role`. This helps describe how the point is used:

- `start`
  - First point in a route or generated pattern.
- `mission`
  - Standard route waypoint.
- `finish`
  - Final point in a route or pattern.
- `turn`
  - Used in generated route structures where a point primarily exists to shape a turn.
- `manual`
  - A user-created route point that was placed directly rather than generated by a pattern.
- `pattern`
  - Not a role itself, but many generated points use `source: pattern` with `start`, `mission`, or `finish` roles.
- `avoid`
  - Not a role itself, but route detour points inserted for boundary avoidance use `source: avoid`.

In practice:

- route behavior is mostly driven by the `nav` settings and leg properties
- `role` is mainly useful for interpretation, export, and downstream logic

## Arrival Actions

Arrival actions are waypoint-level behaviors. They define what happens once a waypoint is reached.

- `Go to Next Waypoint`
  - Continue the route normally.
- `DP Hold`
  - Hold station using dynamic positioning logic.
- `Stop`
  - Stop at the waypoint.
- `Return to Start`
  - Go back to the route start.
- `Complete Reverse Route`
  - Traverse the route back in reverse.
- `Switch to Manual`
  - Hand control back to the operator.
- `Return to Home`
  - Go to the home or recovery reference.
- `Proceed to Next Route`
  - Useful when missions are chained.
- `Hold Then Return`
  - Pause at the point, then return.
- `Loiter Circle`
  - Circle the waypoint area.
- `Await Operator Confirm`
  - Pause until the operator confirms continuation.
- `Go to Recovery Point`
  - Travel to a nominated recovery point.
- `Failsafe Stop`
  - Stop in a conservative failsafe state.

Default behavior:

- intermediate waypoints normally continue to the next point
- the last waypoint may use a different effective default depending on route position logic

## Heading Modes

Heading mode can be applied to both waypoints and legs.

- `Default`
  - Use the planner/export default.
- `Track`
  - Face along the leg bearing.
- `Hold`
  - Keep the current heading.
- `Manual`
  - Heading is expected to be supplied externally or by the operator.
- `Course To Next`
  - Aim toward the next waypoint.
- `Turn Lead`
  - Begin rotating before the waypoint to smooth the turn.

Use `Track` or `Course To Next` for normal route-following. Use `Hold` or `Manual` when the vehicle or tender should maintain a specific orientation for work or observation.

## Movement Modes

Movement mode is leg-level behavior. It defines how the vehicle should travel that segment.

- `Continuous (Default)`
  - Smooth uninterrupted travel along the leg.
- `Intermittent`
  - Move in segments or pauses rather than one continuous run.
- `Seek+Destroy`
  - Aggressive search or contact-style pass behavior.
- `Inchworm`
  - Very deliberate stepwise advancement.
- `Stop-Go`
  - Short move, stop, then continue.
- `Adaptive Speed`
  - Speed varies based on conditions or task.
- `Lawnmower Leg-Scan`
  - Optimised for grid or transect scanning legs.
- `Orbit Track`
  - Track motion intended for orbiting or circling style movement.
- `Contour-Follow`
  - Intended for following a shoreline, reef edge, or shape.
- `Station-Keep Sample`
  - Pause and hold position to capture samples or observations.
- `Current-Compensated`
  - Bias movement to account for current or drift.
- `Energy-Saver`
  - Lower-power transit behavior.
- `Sprint`
  - Faster, more direct travel on the leg.
- `Reacquire`
  - Used when trying to regain position, target, or track.
- `Safety-Crawl`
  - Very conservative slow movement.

These are planner/export intent settings. The exact real-world effect depends on the downstream autonomy or control system that consumes the route.

## Pattern Routing Options

The planner currently supports route pattern insertion from the routing tools and context menu.

- `Spiral Route`
  - Builds a spiral around an anchor.
- `Lawnmower Route`
  - Builds repeated parallel legs for area coverage.

Pattern fields include:

- `Spacing`
  - General spacing between loops or passes.
- `Leg Spacing`
  - Distance between adjacent lawnmower legs.
- `Width`
  - Pattern width.
- `Height`
  - Pattern height.
- `Radius`
  - Spiral radius where relevant.
- `Turns`
  - Number of spiral turns where relevant.
- `Heading`
  - Pattern orientation.
- `Movement Mode`
  - The movement mode applied to created route legs.

Pattern insertion can start:

- from a selected route waypoint
- from a right-click position on the map

The planner can also:

- fill a boundary area with a route pattern
  - especially for lawnmower-style area coverage inside a drawn boundary
  - using the boundary geometry as the area to be surveyed
  - with the resulting route adjusted to stay inside or work around boundary constraints

## Route Selection and Editing

The routing screen supports:

- selecting individual waypoints
- selecting route legs
- selecting the whole route
- moving the whole selected route
- inserting route points on a leg
- reversing the route
- deleting the route from the context menu
- deleting the whole selected route with the `Delete` key

Boundary-aware route movement and pattern generation try to preserve route structure while rerouting around boundary areas.

## Boundary and Avoidance Behaviour

Route avoidance works with:

- boundary legs
- circular boundary areas
- containment-style routing where relevant
- obstacle detours inserted as `avoid` route points

For generated ladder or lawnmower patterns, the planner now aims to:

- keep the main survey legs parallel where possible
- clip legs at boundaries instead of collapsing the whole pattern
- add detour points around the boundary edge
- rejoin the next valid leg after the obstacle

## Reporting and Export Features

The planner has a number of useful reporting and archive features that make it suitable as a project container, not just a route editor.

- `Archive Survey` ZIP export
  - Bundles the current survey into a single archive file.
- `Survey GPX` export
  - Exports the current survey track, sightings, boundaries, ruler, and overlays in GPX-compatible form.
- `Route JSON` export
  - Exports the planner route as machine-friendly JSON for downstream systems such as Node-RED.
- `Planner ZIP round-trip`
  - ZIP archives can now carry explicit planner routing data in `route-mission.json`.
- `Route restore from ZIP`
  - If a loaded archive contains `route-mission.json`, the route waypoints and route leg properties are restored automatically.
- `Overlay round-trip`
  - Overlay GPX files are stored separately inside `overlays/` and reload as overlays instead of being merged into the main survey.
- `Image round-trip`
  - Survey and overlay images can be restored from the archive when present.
- `Backward compatibility`
  - Older planner GPX files that embedded route mission data can still be reloaded, while newer planner archives use ZIP + JSON as the preferred route format.
- `Machine-control export path`
  - The JSON route structure includes waypoint order, roles, nav settings, and per-leg properties for downstream control systems.
- `Project-style archive format`
  - The ZIP format leaves room for future best-of-breed file types to be added without replacing the whole archive structure.

Useful practical outcomes:

- one ZIP can hold the survey track, images, overlays, and route
- the same archive can be reopened later and continue planning work
- the route JSON can be handed to automation or Node-RED without scraping GPX
- GPX remains available for map interoperability and review

## Export Behaviour

Planner route export includes:

- survey metadata
- route waypoint list
- waypoint roles
- waypoint `nav` properties
- route leg properties

The exported route schema is:

- `dropcam.telemetry.planner.route.v1`

## Practical Guidance

Use waypoint properties when:

- you need a specific depth or arrival action at a particular point
- you want point-specific hold or station behavior

Use leg properties when:

- you want a different movement style on one pass
- you want slower or more conservative travel on a difficult section
- you want lawnmower scan behavior only on survey legs

Use heading mode carefully:

- `Track` is the most predictable default for route following
- `Hold` is useful when camera orientation matters more than travel heading
- `Turn Lead` is useful where the vehicle should smooth a corner before reaching the next point

Use movement mode as an intent flag:

- it is useful for downstream automation and reporting
- it does not by itself guarantee a specific vehicle controller implementation unless the downstream system supports it
